  **01**    **프로젝트 개요**

## 1.1 배경 및 목적

당사에서 출원한 타이어 관련 특허에 대해 특허청 심사관으로부터 의견제출통지서를 수령한 상황입니다. 심사관은 선행특허를 근거로 당사 특허의 신규성, 진보성에 이의를 제기하며, 이에 대한 의견서 및 보정서 제출을 요구합니다.

본 프로젝트는 이 심사대응 프로세스를 AI 챗봇으로 자동화, 지원하는 것을 목표로 합니다. 사용자가 의견제출통지서를 입력하면 AI가 분석부터 보정 청구항 초안까지 일련의 흐름을 처리해주는 시스템을 구현합니다.

## 1.2 특허 심사대응 프로세스란?

|**단계**|**명칭**|**설명**|
|---|---|---|
|1|출원|당사가 발명을 특허청에 출원. 청구항(권리 범위), 상세한 설명(기술 내용), 도면 등을 제출.|
|2|심사|특허청 심사관이 선행특허 DB를 검색하여 신규성·진보성 판단.|
|3|의견제출통지|심사관이 거절이유 발견 시 의견제출통지서 발송. 선행특허 번호, 거절 이유, 해당 청구항 명시.|
|4|심사대응 (본 프로젝트)|당사가 통지서를 분석 → 선행특허 비교 → 청구항 보정 또는 의견서 작성.|
|5|등록/거절 결정|심사관이 대응 결과를 재검토 후 등록 또는 최종 거절 결정.|
|6|반복 (복수 차수)|거절이 유지되면 4~5단계 반복. 차수별 변경사항 추적 필요.|

## 1.3 핵심 용어 정리

|**용어**|**설명**|
|---|---|
|청구항 (Claim)|특허의 권리 범위를 정의하는 핵심 문장. 독립항(자립적)과 종속항(독립항 종속)으로 구성.|
|독립항|다른 청구항을 인용하지 않는 청구항. 권리 범위가 가장 넓음.|
|종속항|독립항 또는 다른 종속항을 인용하여 구체화한 청구항.|
|상세한 설명|발명의 기술 내용을 구체적으로 기술하는 부분. 청구항 해석의 기준이 됨.|
|선행특허 (Prior Art)|심사관이 거절 근거로 제시하는 기존 공개 특허.|
|신규성|선행특허에 동일한 발명이 없어야 함.|
|진보성|선행특허로부터 통상의 기술자가 쉽게 발명할 수 없어야 함.|
|보정 (Amendment)|청구항 등을 수정하여 거절이유를 극복하는 행위.|
|Claim Chart|당사 특허 청구항 구성요소와 선행특허를 1:1 대응시킨 비교표.|
|차수|심사대응 회차. 1차 대응, 2차 대응 등으로 관리.|

  

  **02**    **시스템 아키텍처 — Core / Sub-Agent / Tool 계층 구조**

## 2.1 전체 구조 개요

본 챗봇은 Core Agent → Sub-Agent → Tool 의 3계층 구조로 설계됩니다. 각 계층의 역할을 명확히 이해하는 것이 개발의 출발점입니다.

| **계층**     | **역할**                                                                                      |
| ---------- | ------------------------------------------------------------------------------------------- |
| Core Agent | 사용자 쿼리를 이해(NLU)하고, 어떤 Sub-Agent를 어떤 순서로 실행할지 계획(Planning)하며, 최종 응답을 합성(Synthesis)하는 총괄 조율자. |
| Sub-Agent  | 특정 도메인에 특화된 처리 단위. Core Agent의 지시를 받아 Tool들을 조합하여 실행.                                       |
| Tool       | 실제 데이터 처리·분석·생성을 수행하는 최소 실행 단위. 단일 책임 원칙.                                                   |

## 2.2 전체 Sub-Agent 구성 (향후 확장 포함)

현재 확정된 Sub-Agent는 4개이며, 추후 확장 예정입니다.

|**Sub-Agent**|**역할 한 줄 요약**|**본 프로젝트 관여**|
|---|---|---|
|SA-1  Data Agent|특허 데이터 수집 · 조회 (RDB, OpenSearch, MinIO, 외부 DB)|✅ 사전 완료 가정|
|SA-2  Invention Analysis Agent|비정형 텍스트 분석 — 청구항 파싱, 비교, Claim Chart, 보정안 생성|✅ 본 프로젝트 핵심|
|SA-3  Trend Analysis Agent|정형 데이터 통계 · 집계 — 출원 트렌드, 포트폴리오 분석|⬜ 향후 개발|
|SA-4  Output Agent|최종 출력 조합 — 텍스트 · 표 · 차트 · 문서 생성|⬜ 향후 개발|

**📌** **본 프로젝트 범위:** SA-1(Data Agent)은 당사 특허 및 선행특허의 청구항·상세한 설명 데이터를 이미 가져온 상태로 가정합니다. 본 프로젝트는 SA-2 Invention Analysis Agent 내 심사대응 기능 구현에 집중합니다.

## 2.3 데이터 흐름

|**단계**|**주체**|**입력**|**출력**|
|---|---|---|---|
|0|SA-1 (가정)|특허번호, 통지서|청구항 텍스트, 상세한 설명 텍스트|
|1|Core Agent|사용자 자연어 쿼리|SA-2 실행 계획|
|2|SA-2 Tool 1|의견제출통지서 텍스트|심사관 거절이유 구조화 JSON|
|3|SA-2 Tool 2|청구항 텍스트|파싱된 청구항 구조체|
|4|SA-2 Tool 3|당사·선행 청구항 구조체 + 상세한 설명|Claim Chart|
|5|SA-2 Tool 4|Claim Chart + 거절이유|차이점 분석 + 대응 전략|
|6|SA-2 Tool 5|차이점 분석|보정된 독립항 초안|
|7|SA-2 Tool 6|차수 메타데이터|버전 저장 · 변경사항 추적|

**🔑** **개발 원칙:** Tool 개발이 선행되어야 Agent 개발이 가능합니다. 각 Tool은 독립적으로 단위 테스트 가능한 함수 단위로 구현하며, Tool이 완성된 후 SA-2 Agent가 이를 조합하여 실행합니다.
